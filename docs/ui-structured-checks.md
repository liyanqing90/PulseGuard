# UI 结构化校验构思

> 历史草案说明：当前主线采用“前置脚本 + 页面扫描候选 + 结构化断言”。录制器不进入当前主线；以下内容仅保留结构化断言方向，涉及录制的描述视为远期观察项。

目标：让 UI 任务像接口任务一样，优先通过可视化操作完成配置，脚本只作为高级模式保留。

## 推荐交互

1. 发起页面预览
   - 输入 URL 后点击“扫描页面”。
   - 系统启动 Playwright 预览页，采集页面标题、URL、可见文本、可点击元素、输入框、按钮、链接和截图。

2. 扫描候选并生成断言草稿
   - 系统先执行 `setup_script`，前置失败时中止并返回脱敏原因。
   - 页面扫描返回标题、URL、可见文本、可点击元素、输入框、按钮、链接和稳定选择器候选。
   - 用户从候选中选择断言草稿，当前不建立完整操作流。

3. 点选元素添加校验
   - 用户在预览页点选元素。
   - 右侧显示元素名称、角色、文本、稳定选择器候选、可见状态。
   - 可一键添加校验：元素可见、元素隐藏、文本包含、文本等于、元素数量、属性值、输入框值、按钮可用、URL 包含、标题包含。

4. 调试结果内联展示
   - 不跳新页面，编辑器底部直接展示检测结果。
   - UI 结果优先展示：断言列表、每项状态、失败断言、截图和必要的控制台摘要。
   - 失败时自动定位到失败断言和对应页面证据。

5. 高级模式
   - 保留 Python 脚本折叠区。
   - 结构化断言存在时默认不要求脚本。
   - 需要复杂登录、多窗口、业务分支时再使用脚本。

## 数据结构草案

```json
{
  "setup_script": "async def setup(ctx):\n    page = await ctx.new_page()\n    await page.goto(ctx.entry_url)\n    return page\n",
  "assertions": [
    {"type": "element_visible", "selector": "text=首页"},
    {"type": "text_contains", "selector": "[data-testid='user-name']", "expected": "张三"},
    {"type": "url_contains", "expected": "/dashboard"}
  ]
}
```

## 第一阶段实现切片

- 继续使用 `setup_script` 和 `assertions_json`。
- UI 编辑器增加“扫描页面”和“添加校验项”。
- 先支持 6 个规则：页面标题包含、URL 包含、文本出现、元素可见、元素隐藏、元素数量。
- 检测结果用表格展示：规则、目标、匹配数、状态、耗时、错误。
- 高级脚本继续保留，结构化 UI 校验存在时不要求脚本。
